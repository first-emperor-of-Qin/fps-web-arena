## 第三轮：端到端验证（本轮）
- 目标：确认「关卡联机 + 王者乱斗」同步到底通不通，定位「完全无法同步」真因。
- 测试方法（均指向隔离测试服 localhost:3100，Node 26 起，不动用户 3000 现场）：
  1. **服务端中继层** Node `ws` 裸连测试：PV(pve_create/join/start/sync 广播) + 王者乱斗(match_found 配对 + snap 广播 + hit 转发) 共 10/10 通过 → 中继层本身无问题。
  2. **真实双 Chrome(headless swiftshader) UI 流程**：
     - 关键坑：给 `window.WebSocket` 套探针会**破坏游戏自身自动连接**（auto-connect 在重载瞬间被包裹打断）→ 改用「引导启动完成后再挂 `window.net.send` 猴子补丁 + 给 `window.net.ws` 加 message 监听」来抓流量，不碰游戏连接。
     - 王者乱斗 1v1：双方 `match_found`、各自建远程玩家、A 上报 167 帧 snap / B 收 168 帧且存 6 帧快照 → **位置同步完全可用**。
     - PVE：建房/加入/开始 均通；A 真实击杀→广播 `pve_sync`(A_sent=1)→B 收到(B_recv=1)→B 击杀列表 +1(应用到账)。**收发+应用链路通**。
- **结论**：两种模式的联机同步在「网络 + 应用」两层**均确认可用**，架构（客户端各自模拟 + 服务端仅中转）完全支撑联机，技术上可行。
- **「完全无法同步」最现实的根因 = 主循环门控 `!isPanelOpen` 卡死**：若进组队前开过任意面板（其 DOM 仅 `display:none` 不复位 `isPanelOpen`），主循环门控为假 → 整局不刷怪/不结算/不收快照 → 看起来「什么都不同步」。已在 `startOnlineMatch`(30301) 与 `__startPveCoop`(30439) 显式 `isPanelOpen=false` 复位（前两轮未修此点）。
- 补充：`createEnemy→finalizeEnemy` 创建战役敌人**不带 `isBot` 字段**（值为 undefined，falsy），故 `killEnemy`/`__pveOnEnemyKilled` 的 `if(enemy.isBot) return` 不会误伤战役敌人，广播正常。此前误判「isBot 拦截」已排除。
- 临时验证用的 `window.__pveDebug` 钩子已移除，文件无调试残留。

## 第四轮：PVE 组队大厅 4 个交互 bug 修复（本轮）
用户反馈 4 个组队大厅问题，全部定位并修复，双浏览器/服务端双重验证通过。
### Bug1 邀请好友按钮"有时无响应"
- 根因：`renderPveInviteFriends` 用 `MP.friends.filter(f=>f.online)` 只显示"在线"好友。好友在线状态由 WS `connsByUser` 决定；若两人虽都在游戏内但 presence 推送有瞬间延迟/或好友被标记离线，弹窗就只显示"暂无在线好友" → 看起来"点了没反应"。
- 修复：邀请弹窗改为**展示全部好友**（在线高亮"● 在线"，离线显示"○ 离线（上线后可见邀请）"），邀请按钮对所有人可点。只要对方真在游戏内（WS 已连）就能收到邀请。
### Bug2 队长邀请后邀请列表不实时显示该好友
- 根因：服务端只在对方"接受"后才 `roster` 广播；队长侧没有"待接受邀请"状态，故发出邀请后大厅看不到谁已被邀。
- 修复：服务端 `room.invited`(Map) 记录已邀请未接受者；`pve_invite` 广播 `pve_invites`(userId 列表) 给房间内；`pve_accept_invite` 接受后 `delete` 并再广播清空。客户端新增 `MP.onPveInvites` + 大厅渲染"待接受邀请"区（`#pve-invites`）。
### Bug3 关卡不同步（队长选5关，被邀请人显示1关；切换也不变）
- 根因：① 关卡 `<select>` 没有任何 `onchange` 处理，队长切关后**只改了本地 DOM，从不上报服务端** → `room.levelNo` 永远是建房时的默认值 1；② `roster` 只更新玩家不更新关卡；③ 邀请消息带的是服务端旧 `levelNo`。所以被邀请人永远看到第1关。
- 修复：① `#pve-level-select` 加 `onchange`（仅队长）→ 发 `pve_set_level`；② 服务端新增 `pve_set_level` case，更新 `room.levelNo` 并 `broadcastRoom(pve_level)` + 对 `room.invited` 中待接受者逐一 `sendToUser(pve_level)`；③ 客户端新增 `MP.onPveLevel`：非队长时把 `#pve-level-select` 值同步为服务端关卡（含邀请弹窗未接受时的关卡显示）。`renderPveLobby` 也按 `MP.pveRoom.level` 校正。
### Bug4 组队界面关卡列表不匹配（仅5个且第5关名错）
- 根因：`<select id="pve-level-select">` 硬编码只有 5 个 `<option>`，且第5关误写为"终焉之城"（实际第5关是"毒雾沼泽"）；而游戏实际有 10 关。

### ② BOSS 击败不同步——4 项根因彻底重构（v8）
**现象**：组队时 A 端已显示通关（showVictoryScreen），B 端仍在与 BOSS 战斗。

**根因分析（4 层叠加）**：
1. **[BUG] `kind:'boss'` 处理逻辑反了**：原代码先搜"任意存活非 bot 敌人"再兜底搜 BOSS → 战斗中若还有小怪存活，BOSS 死亡消息会误杀一个小怪而非 BOSS，真正的 BOSS 继续活着。
2. **无权威强制置零**：远端报"BOSS 已死"，本地收到后只对找到的目标调用 `killEnemy()`，但不会强制把本地 BOSS 血量置 0。若搜索命中了错误目标或遗漏，本地 BOSS 仍满血战斗。
3. **仅增量 dmg 同步无全量校准**：每条伤害消息独立中继，丢包=伤害永久丢失；长时间 BOSS 战斗（HP 亿级）累积漂移可达数百万→两端击杀时刻差异巨大。
4. **BOSS 死亡通知仅发一次**：网络抖动时单条消息丢失 = 永久不同步（无重试/无补偿机制）。

**修复方案（4 层对应 4 项修复）**：

#### Fix A: `onPveSync` boss/kill 分离 + BOSS 优先精确匹配
- 原：`if(d.kind==='kill'||d.kind==='boss'){ 搜任意敌人; if(!enemy && boss) 兜底BOSS }`
- 新：
  - `kind:'boss_hp'` → 全量校准分支（见 Fix C）
  - `if(d.kind === 'boss'){ 先搜 be.isBoss(未同步) → 兜底搜任意 isBoss(含已同步) → 强制 health=0 → killEnemy → return }`
  - `if(d.kind === 'kill'){ 原逻辑不变(搜任意敌人) }`
  - boss 分支有 `return` 阻断，绝不落入 kill 逻辑

#### Fix B: 权威强制置零
- 收到 `kind:'boss'` 且找到 bTarget 后：`if(bTarget.health > 0){ bTarget.health = 0; updateEnemyHealthBar(bTarget); if(bTarget===boss) updateBossBar(); }`
- 无论本地 BOSS 当前血量多少，远端说死了就必须死。

#### Fix C: 每 2s 全量 BOSS HP 校准广播（`kind:'boss_hp'`）
- 发送端（在现有 __pveDmgInterval 1s 定时器内，新增 _bossHpTick 计数器，每 2 tick 即 2s 触发一次）：
  ```
  MP.send({type:'pve_sync', data:{ kind:'boss_hp', uid:boss._uid, hp:Math.round(boss.health) }})
  ```
  仅在 `boss && !boss._pveKillSynced && boss._uid` 时发送。
- 接收端（onPveSync 新增 boss_hp 分支）：
  - `remoteHp = Math.max(0, Number(d.hp))`
  - `if(remoteHp < boss.health)` → `boss.health = remoteHp` + updateBossBar() + updateEnemyHealthBar()（取 min，保守趋同）
  - `if(remoteHp <= 0 && boss.health > 0)` → 强制 health=0 + killEnemy（权威覆盖）

#### Fix D: BOSS 击杀重发机制
- `__pveOnEnemyKilled` 中：BOSS 击杀时 payload 对象化，首次发送后在 1s/2s 各补发一次（共 3 次）。
- 仅 `isBoss` 时触发重发（普通击杀不重发避免流量浪费）。
- 重发前检查 `pveCoopActive` 仍有效（防止跨局残留重发）。
- payload 用 `JSON.parse(JSON.stringify(payload))` 深拷贝防止闭包引用污染。
